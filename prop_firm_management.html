‚ö° Copier Integration Layer

The trade copier is the execution layer that replicates validated trades across multiple prop firm accounts. It must be treated as a pluggable module so that different vendors (e.g., TradeSyncer, Rithmic-native, Tradovate/NinjaTrader copiers) can be swapped without changing the compliance or strategy logic.

Integration pattern

  1. Flow: TradingView ‚Üí AWS Compliance Middleware ‚Üí Copier Adapter ‚Üí Prop Firm Accounts
  2. Guarantee: Only trades that pass compliance checks are forwarded to the copier.
  3. Abstraction: Define a standard Copier API payload; implement vendor-specific adapters.

Standard Copier API payload

{
  "action": "placeOrder",
  "accountId": "APX-123456",
  "symbol": "NQ1!",
  "direction": "long",
  "qty": 2,
  "entry": 15678.25,
  "stop": 15658.25,
  "takeProfit": 15718.25,
  "strategyId": "liq_sweep_momo_v3",
  "timestamp": "2025-09-28T04:30:00Z"
}

Vendor adapter examples

  • TradeSyncer: Cloud API; supports Apex, Bulenox, Take Profit Trader. Adapter translates standard payload into TradeSyncer API calls.
  • Rithmic-native copier: Ultra-low latency; adapter must handle Rithmic session auth and order placement.
  • Tradovate/NinjaTrader copiers: Adapter wraps their APIs or local connectors.

Risk guardrails at copier level

  • Lot multipliers: Scale trades differently per account.
  • Max contracts: Enforce hard caps per account.
  • Equity-based scaling: Adjust trade size proportional to account balance.

Monitoring and health checks

  • Heartbeat: Lambda scheduled check against copier API; update dashboard with status (Online/Offline).
  • Execution logs: Every copier action (accepted, rejected, delayed) is written to DynamoDB Trades with ExecutionStatus.
  • Alerts: CloudWatch alarm if copier API fails or latency exceeds threshold.
